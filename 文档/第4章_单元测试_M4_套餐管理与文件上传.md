# 第4章 单元测试

## 4.4 套餐管理与文件上传模块单元测试（M4）

### 4.4.1 测试概述

#### 4.4.1.1 测试范围与目标

本模块单元测试覆盖苍穹外卖系统中套餐管理子系统和文件上传功能的全部核心代码，具体包括以下五个文件的测试：

| 被测文件 | 所属层级 | 职责描述 |
|----------|----------|----------|
| `SetmealController.java` | Controller层 | 管理端套餐CRUD接口（新增、分页查询、删除、根据ID查询、修改、启停共6个接口） |
| `CommonController.java` | Controller层 | 通用文件上传接口（`/admin/common/upload`） |
| `SetmealServiceImpl.java` | Service层 | 套餐业务逻辑实现，含起售套餐的菜品状态校验、删除套餐的在售状态校验等核心业务规则 |
| `SetmealMapper.java` | Mapper层 | 套餐表数据库操作（增删改查） |
| `SetmealDishMapper.java` | Mapper层 | 套餐-菜品关联表数据库操作（批量插入、按套餐ID查询、按套餐ID删除） |

测试目标如下：

1. **Controller层验证**：通过MockMvc模拟HTTP请求，验证各接口对正常输入和异常输入的正确响应，确保返回正确的HTTP状态码和JSON响应体结构。
2. **Service层验证**：通过Mockito模拟Mapper依赖，重点验证起售套餐时菜品状态校验逻辑的正确性，确保在包含未启售菜品时正确抛出`SetmealEnableFailedException`异常，并验证其他业务方法的正常执行路径。
3. **Mapper层验证**：通过`@MybatisTest`结合H2内存数据库，验证SQL语句的正确性和数据持久化行为，包括套餐的增删改查以及套餐-菜品关联关系的插入、查询和删除操作。
4. **用例设计方法对比**：分别采用判定表法和等价类划分法对同一被测功能（起售套餐业务逻辑）设计测试用例，在4.4.5节进行两种方法的比较分析。

#### 4.4.2.3 Controller层测试过程中的差异情况

本次Controller层测试未发现预期与实际结果之间的重大差异。以下仅记录测试执行过程中的若干观察：

1. **SetmealController新增接口测试（TC-SC-002至TC-SC-004）**：由于Controller层测试采用Mock Service的方式，名称为空、价格为负数、套餐菜品为空等边界异常数据在Service层Mock中直接返回`void`，因此Controller层均返回`code=1`的成功响应。实际业务校验逻辑需在Service层或参数校验层进行拦截，后续测试阶段应关注参数校验注解（如`@NotBlank`、`@NotNull`等）是否已正确地配置于DTO类中。

2. **CommonController文件上传文件名空字符串测试（TC-CC-003）**：测试过程中发现，当上传文件名为空字符串时，Controller层代码会自动生成默认文件名并使用`.jpg`作为默认后缀，因此接口仍能成功执行并返回文件URL。这一行为属于预期的容错处理机制。

3. **CommonController无文件测试（TC-CC-007）**：当请求中不包含任何文件时，接口返回`code=0`且提示消息为"文件不能为空"，验证了文件非空校验逻辑的正确性。该用例是Controller层29个测试用例中唯一一个预期`code`不等于1的用例。

1. **`startOrStop`方法的设计模式**：源代码中起售分支采用"先检后改"模式，在事务边界内先查询菜品状态，若发现未启售菜品则立即抛出业务异常，阻止后续的状态更新操作。这种设计确保了数据一致性，但也意味着起售失败时不会对套餐状态产生任何修改，符合事务一致性的要求。

2. **`deleteBatch`方法的级联删除逻辑**：删除套餐时遵循"先删关联，再删主表"的顺序，确保外键约束不会导致删除失败。同时，在删除关联记录之前先校验套餐状态，避免了在售套餐被误删的风险。

3. **Mock验证的粒度**：Service层测试仅验证业务逻辑的执行路径是否正确（是否抛出异常或不抛出异常），不验证Mapper方法的具体调用参数，符合单元测试的"测行为不测实现"原则。

### 4.4.5 测试用例设计方法比较分析

根据测试人员分工要求，本模块在单元测试设计过程中分别采用了**判定表法**和**等价类划分法**对同一被测功能进行用例设计，并在本节对两种方法进行比较分析。选取的对比对象为`SetmealServiceImpl.startOrStop()`方法（起售套餐时的菜品状态校验逻辑），该方法包含复杂的条件组合和业务规则，适合作为方法对比的典型场景。

#### 4.4.5.1 判定表法设计用例

**被测功能描述**：`startOrStop(Integer status, Long id)` —— 当status为ENABLE(1)时，需检查套餐关联的所有菜品状态；当status为DISABLE(0)时，不检查菜品直接更新状态。

**条件桩与动作桩分析**：

| 条件 | 取值 |
|------|------|
| C1：目标状态(status) | 1 (ENABLE) / 0 (DISABLE) |
| C2：菜品列表是否为null | 是 / 否 |
| C3：菜品列表是否为空(size=0) | 是 / 否 |
| C4：菜品列表中是否存在状态为DISABLE的记录 | 是 / 否 |

| 动作 | 描述 |
|------|------|
| A1：更新套餐状态 | 执行`setmealMapper.update()` |
| A2：抛出SetmealEnableFailedException | 拒绝启售 |

**简化判定表（合并规则后）**：

| 规则编号 | C1(目标状态) | C2(列表null) | C3(列表空) | C4(含未启售菜品) | 预期动作 |
|----------|-------------|-------------|-----------|-----------------|----------|
| R1 | 1 (ENABLE) | N | N | N | A1：更新套餐状态为启用 |
| R2 | 1 (ENABLE) | N | N | Y | A2：抛出SetmealEnableFailedException |
| R3 | 1 (ENABLE) | N | Y | — | A1：更新套餐状态为启用（列表空，无菜品可检查） |
| R4 | 1 (ENABLE) | Y | — | — | A1：更新套餐状态为启用（列表null，无菜品可检查） |
| R5 | 0 (DISABLE) | — | — | — | A1：更新套餐状态为停用（不检查菜品） |

**依据判定表生成的测试用例**：

| 对应规则 | 测试用例对应方法 | 测试描述 |

#### 4.4.5.3 两种方法的比较分析

以下从多个维度对判定表法和等价类划分法进行系统比较：

| 比较维度 | 判定表法 | 等价类划分法 |
|----------|---------|-------------|
| **核心思想** | 将条件与动作以表格形式列出，穷举条件组合对应的动作，适合多条件组合场景 | 将输入域划分为若干等价类，每个等价类中任意值代表一类输入，适合输入空间较大的场景 |
| **适用场景** | 条件之间存在逻辑组合关系，不同条件组合产生不同行为输出 | 输入参数为连续值域或离散值集合，行为差异主要由输入分类决定 |
| **用例完备性** | 通过穷举条件组合确保逻辑覆盖完整，不易遗漏条件交互产生的边界场景 | 仅保证每个等价类至少覆盖一次，可能遗漏等价类之间的交互情况 |
| **用例数量** | 5条化简规则，实际扩展为7条 | 5个等价类，实际扩展为7条 |
| **逻辑可视化** | 判定表形式直观展现条件间的AND/OR逻辑关系，便于发现缺失的规则 | 等价类列表较为简洁，但对于多条件交互场景不够直观 |
| **冗余控制** | 可通过化简合并减少冗余规则（如本案例中R3/R4可进一步合并） | 每个等价类通常仅需1个代表值，天然控制冗余 |
| **对本被测功能的适用度** | ⭐⭐⭐⭐⭐ 非常适合。本功能涉及4个条件的组合判断，判定表能清晰展示"status=ENABLE且菜品非null非空且无未启售菜品→正常启用"等复杂组合逻辑 | ⭐⭐⭐ 基本适用。等价类能有效划分输入，但无法直观展示status与菜品状态之间的AND关系 |
| **缺陷发现能力** | 较高。R4规则（菜品列表为null时正常启用）在实际代码中对应一个容易遗漏的防御性判断`if(dishList!=null)`，通过判定表穷举时不易遗漏 | 中等。等价类法若仅划分有效/无效等价类，可能遗漏"null列表"这一特殊边界场景（null与空列表的区分） |

**方法选择的综合建议**：

对于`startOrStop`这类条件组合明确、业务规则以"条件→动作"形式表达的场景，判定表法更适合作为主要设计方法。它在逻辑完整性和缺陷发现能力方面表现更优，能够系统性地覆盖所有条件排列组合以及条件之间的交互情况。等价类划分法更适合输入维度单一或输入空间连续的场景（如本模块中文件上传的大小边界测试TC-CC-004至TC-CC-006），或作为判定表的补充来验证边界条件。

本次测试实践中，最终综合采用两种方法互补设计：以判定表法为主框架穷举所有条件组合，以等价类划分法的思想对边界情况进行补充验证，确保测试用例的完备性和有效性。
