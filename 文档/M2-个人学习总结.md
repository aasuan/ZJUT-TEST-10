# M2 个人学习总结

> 角色：M2（员工管理 + JWT 管理端鉴权）  
> 篇幅：约 480 字

---

本次苍穹外卖测试项目中，我负责员工管理与 JWT 鉴权模块，从零完成了 Controller、Service、Mapper 三层单元测试，以及基于 Docker MySQL/Redis 的集成测试，对 Spring Boot 项目的分层测试有了较系统的实践体会。

在单元测试阶段，我按「由外到内」的顺序推进：Controller 层用 `@WebMvcTest` 和 MockMvc 验证 HTTP 响应与 JSON 结构，并 Mock Service 隔离业务；Service 层用 Mockito 模拟 Mapper，重点覆盖 login 的成功、账号不存在、密码错误、账号锁定四条分支；Mapper 层用 `@MybatisTest` 配合 H2 内存库，验证 SQL、唯一约束和分页条件。这一过程中，我体会到单元测试的核心是**缩小范围、精确隔离依赖**，否则一个小改动就会牵动整条链路。

集成测试则让我看到单元测试的边界：只有连接真实数据库、启动完整 Spring 上下文，才能验证 JWT 签发与 `JwtTokenAdminInterceptor` 的 401 拦截。期间遇到 integration 配置与 H2 脚本冲突、Docker 未启动等问题，通过调整 `application-integration.yml`、用 `@Sql` 注入种子数据逐一解决，也加深了对「测试环境」与「被测系统」边界的理解。

用例设计方面，我以 login 接口为例，对比了等价类划分与边界值分析：等价类适合快速覆盖四类业务结果，边界值则补充密码相邻错误、账号禁用等临界场景。实践中二者结合效果最好——W2 保证分支逻辑，W3 补全鉴权链路。

不足与收获：一是对空用户名、空密码等参数边界尚未单独建用例，认识到测试也要记录「已知缺口」而非只写通过用例；二是 Git 协作、分支命名、测试目录 gitignore 等工程细节，与写测试代码同样重要。今后希望在系统测试与性能配合中，继续把「设计用例—编写自动化—分析结果」串成完整闭环。
