## 第1章 范围

### 1.1 项目背景

苍穹外卖是一个基于 Spring Boot 开发的在线外卖订购系统，采用前后端分离架构。系统支持管理端（菜品管理、套餐管理、订单管理、员工管理、数据报表）和 C 端（用户浏览、购物车、下单、支付）两大业务场景。

| 项目 | 说明 |
|---|---|
| 代码仓库 | `https://github.com/Sonder-MX/sky-take-out.git` |
| 技术栈 | Spring Boot 3.1.2 + MyBatis + MySQL 8.0 + Redis 7 |
| 模块结构 | sky-common / sky-pojo / sky-server |
| 现有测试基础 | 零，需从零搭建 |

### 3.2.4 性能测试结果分析

#### 整体 TPS-并发数据

| 并发数 | 整体 TPS | 较上一级变化 | 变化率 |
|---|---|---|---|
| 50 | 4,863.7 | — | — |
| 100 | 4,986.3 | +122.6 | +2.5% |
| 200 | **5,106.1** | +119.8 | +2.4% |
| 500 | 4,754.7 | -351.4 | **-6.9%** |

![TPS-并发曲线图](../压测相关/tps_concurrency_curve.png)

**Fig.3-1：混合场景 TPS-并发曲线图**

#### 分析结论

**（1）50→200 并发：扩展性良好**

TPS 从 4,863.7 稳步增长至 5,106.1，累计增幅约 5%。系统在 200 并发以内吞吐能力随负载线性提升，说明 Spring Boot 默认线程池（Tomcat 200 线程）和 HikariCP 连接池在此范围内工作正常。

**（2）200→500 并发：出现性能拐点**

500 并发时 TPS 降至 4,754.7，较峰值下降 6.9%。TPS 不升反降的典型拐点信号说明：
- 请求开始排队等待，线程上下文切换开销超过并行处理收益
- 数据库连接池（HikariCP 默认 10 连接）饱和，写操作等待连接释放
- JMeter 与服务器同机运行，CPU 资源争抢加剧

**（3）性能评估**

| 指标 | 数值 | 评估 |
|---|---|---|
| 峰值 TPS | 5,106.1 | 满足小型外卖系统需求（日均千单级） |
| 最佳并发 | 200 | 建议生产环境控制在此范围内 |
| 500 并发退化 | -6.9% | 明确天花板，需优化后方可扩容 |

### 3.2.5 性能瓶颈定位与优化建议

#### 瓶颈定位（按可能性排序）

| 排序 | 瓶颈点 | 依据 | 严重程度 |
|---|---|---|---|
| 1 | HikariCP 连接池（默认 10） | 下单接口 S7 需要写库，连接池耗尽导致等待 | 🔴 高 |
| 2 | Tomcat 线程池（默认 200） | 超过 200 并发请求进入 Accept 队列排队 | 🔴 高 |
| 3 | JMeter 与服务器同机资源争抢 | JMeter 500 线程自身消耗大量 CPU | 🟡 中 |
| 4 | MySQL 写锁竞争 | S7 下单涉及多表 INSERT，高并发下锁等待 | 🟡 中 |
| 5 | Redis 连接池不足 | S3/S4 缓存查询高并发下 lettuce 连接耗尽 | 🟢 低 |

#### 优化建议

| 优先级 | 优化项 | 配置/方案 | 预期效果 | 实施难度 |
|---|---|---|---|---|
| 🔴 高 | HikariCP 扩容 | `spring.datasource.hikari.maximum-pool-size=30` | 缓解连接池瓶颈 | 低 |
| 🔴 高 | Tomcat 扩容 | `server.tomcat.threads.max=400` | 减少请求排队 | 低 |
| 🟡 中 | 分机压测 | JMeter 与服务器分机部署 | 消除资源争抢 | 中 |
| 🟡 中 | Redis 缓存预热 | 启动时加载高频菜品/套餐数据 | 提升 S3/S4 TPS | 中 |
| 🟢 低 | 下单异步化 | RocketMQ/Kafka 削峰填谷 | 提升 S7 抗压能力 | 高 |

#### 总体评价

系统在当前配置下峰值 TPS **5,106**，200 并发以内性能稳定，满足课程设计级别的小型外卖系统性能要求。若需支撑更高并发，扩容连接池和线程池即可获得立竿见影的提升效果。

**测试环境**：Windows 11 + JDK 17 + JUnit 5 + Mockito 5.x；Controller `@WebMvcTest` + MockMvc，Service `@ExtendWith(MockitoExtension.class)`，Mapper `@MybatisTest` + H2。

#### 4.4.1 Controller 层

| 被测类 | 接口数 | 用例数 | 关键覆盖 |
|--------|--------|--------|----------|
| SetmealController | 6 | 22 | 新增（空名称/负价格/空菜品列表边界）、分页（pageSize=1/100 边界）、启停（status=2 非法值） |

#### 4.4.2 Service 层（14 个用例）

重点验证 `startOrStop()` 的菜品状态校验：
- `status=ENABLE` + 全部菜品启售 → 正常启用
- `status=ENABLE` + 存在未启售菜品 → 抛出 `SetmealEnableFailedException`
- `status=ENABLE` + 菜品列表为空/null → 正常启用（防御性容错）
- `status=DISABLE` → 不检查菜品状态，直接停用

同时覆盖 `deleteBatch`（在售套餐不可删除→`DeletionNotAllowedException`）、`saveWithDish`、`update`、`getById`、`pageQuery` 等业务方法。

## 第8章 社会可持续发展

软件测试作为软件开发生命周期中的关键环节，其价值不仅体现在保障单个项目的质量上，更对社会可持续发展具有深远影响。本章从可靠性保障、资源节约、行业生态和人才培养四个维度，阐述苍穹外卖系统测试实践对社会可持续发展的贡献。

### 8.1 保障软件可靠性，降低社会运行风险

随着外卖、出行、医疗等生活服务类软件深入社会运转的毛细血管，软件缺陷可能导致的后果已远超"功能不可用"的范畴。一个外卖系统的订单提交失败可能导致商户损失收入、骑手空跑、用户等待超时——这些连锁反应最终转化为社会资源的浪费和信任成本的上升。

本项目通过 **355 个单元测试用例**和 **32 个集成测试场景**，在开发早期识别并修复了包括购物车数据隔离失效、MyBatis 参数绑定错误、日期循环逻辑错误等在内的 5 个正式缺陷。其中 BUG-002（MyBatis `<foreach>` 参数绑定错误）为致命级阻塞缺陷——如果未被发现而上线，将导致所有 C 端用户无法提交订单。这类缺陷在生产环境中每存在一分钟，都会对社会经济活动造成实际影响。测试工作将这些风险消除在交付之前，是对社会负责任的表现。

### 8.2 性能测试引导资源优化，践行绿色计算理念

性能测试的核心产出——TPS-并发曲线和瓶颈分析——不仅是技术指标，更具有资源优化的社会意义。本项目的 JMeter 压力测试揭示了系统在 200 并发处的性能拐点：当并发从 200 提升至 500 时，TPS 反而下降了 6.9%，表明服务器已进入"空转耗能"状态——CPU 大量消耗在线程切换和连接等待上，而实际业务吞吐反而降低。

这一发现促使我们提出了明确的优化建议（HikariCP 连接池扩容、Tomcat 线程池调整、Redis 缓存预热），旨在让系统在相同硬件条件下处理更多业务请求。在更大尺度上，如果每个软件系统都经过充分的性能调优，全球数据中心的能耗将显著降低——这正是"绿色软件工程"（Green Software Engineering）的核心主张。本项目的性能测试实践，虽规模有限，却为团队成员植入了"性能即可持续性"的工程意识。

### 8.3 自动化测试与 CI/CD 降低长期维护成本

手工测试的重复执行不仅效率低下，更不可持续——每次代码变更都要人工回归验证，人力资源瓶颈最终会导致测试覆盖的妥协，从而埋下质量隐患。本项目从零搭建了完整的自动化测试体系：

- **单元测试**：37 个测试类通过 GitHub Actions CI 在每次 Push 时自动执行，无需人工介入
- **集成测试**：`@SpringBootTest` + Docker 中间件，实现真实环境的自动化全链路验证
- **API 自动化**：Apifox 覆盖全部 62 个接口端点

这些自动化资产将在项目的整个生命周期中持续发挥作用。规范化测试体系的可迁移性意味着后续项目可以直接复用测试架构（JUnit5 + Mockito + H2 + CI 流水线），降低软件行业的整体"重复造轮子"成本。

### 8.4 Jira 项目管理促进工程规范化与知识传承

通过 Jira 的 Epic → Task → Bug 三级管理体系，本项目实现了测试全过程的数字化和可追溯。每个 Task 的 Given-When-Then 描述、Review 审查记录、Bug 的复现步骤和修复方案，构成了完整的测试知识库。这套规范化的工程管理方法本身就是可持续的——新成员加入时可以快速通过 Jira 历史了解项目的测试策略和已知问题，避免知识孤岛的形成。

总之，苍穹外卖系统的测试实践表明：软件测试不仅是一道质量防线，更是一种对社会资源负责的工程态度。从预防缺陷导致的社会经济损失，到性能优化带来的能源节约，再到自动化测试降低的长期人力成本——测试工作在每个层面都与社会可持续发展同频共振。

## 第9章 个人学习总结

本章收录了苍穹外卖测试团队 8 名成员的个人学习总结。每位成员回顾了自己在为期 4 周的测试实践中掌握的方法、遇到的挑战以及由此产生的对大型软件工程开发的认知深化。

### 9.1 M1（组长）— 从零搭建测试体系的全局视野

作为组长，我的核心任务不是测试某一个模块，而是搭建让 7 名组员能够高效协作的测试基础设施。这个角色让我深刻体会到：**大型项目的成败不仅取决于个人技术能力，更取决于工程体系的完善程度**。

在项目管理层面，我通过 Jira 建立了 Epic → Task → Bug 三级管理体系，将测试工作拆解为可追踪、可审查的原子单元。每天 Review 组员的测试代码让我意识到"规范先行"的重要性——统一的命名规范（如 `[模块]-[编号] [层级]: 描述`）和 Description 模板（Given-When-Then）在初期看似额外负担，却在 W3-W4 的报告整合阶段发挥了巨大价值：我可以直接从 Jira 导出完整的数据，而不需要向 7 个人逐个追问。

在 DevOps 层面，搭建 GitHub Actions CI 流水线和 Docker 测试环境是我最重要的技术收获。CI 的每一次自动运行都是对"代码质量不应依赖人工记忆"这一工程原则的实践。Docker Compose 一键启动 MySQL + Redis 测试环境，消除了"在我机器上能跑"的环境差异问题。这些看似与"测试"无关的基础设施工作，恰恰是大型项目能够持续迭代的前提条件。

这次实践彻底改变了我对"测试"的认知：测试不是一个阶段，而是一种贯穿开发全程的工程文化。测试金字塔（单元→集成→系统）、CI/CD 自动化、Jira 项目管理——这些方法共同构成了一个"质量内建"的体系，而非"事后检查"的工序。

### 9.2 M2 — 从 Mock 到真实：理解测试分层与等价类边界值

我负责员工管理与 JWT 认证模块，涵盖 `EmployeeController` 的 7 个接口和 `EmployeeServiceImpl` 的登录业务逻辑。这个模块看似简单，却让我系统地学习了三层测试架构（Controller → Service → Mapper）以及两种经典的测试用例设计方法。

**等价类划分法**让我学会从业务结果出发划分输入空间。以登录接口为例，4 个等价类（成功/不存在/密码错/锁定）精确对应 4 条核心用例，保证分支不遗漏。**边界值分析法**则在此基础上发现了"密码差一位"这类相邻错误的独特价值——这种"差一点就对"的场景在等价类框架下容易被忽略，却是真实用户最常遇到的输入错误。

在技术层面，`@WebMvcTest` 切片测试中 `GlobalExceptionHandler` 未自动加载的问题，让我理解了 Spring Boot 测试上下文的加载机制；MockMvc 的请求构建和 JSON 断言则让我掌握了 HTTP 层测试的标准范式。M2 测试方法对比的结论在实际开发中具有指导意义：**等价类适合作为主框架保证逻辑分支完备，边界值用于补充输入端临界场景**。

### 9.3 M3 — 复杂业务逻辑的分层测试与因果图法的威力

M3 模块是本项目中文件数量和接口数量最多的模块之一（7 个被测文件、118 个测试用例），覆盖分类管理和菜品管理两大业务域。这次实践让我对复杂业务系统的测试策略有了系统性认识。

**场景法与因果图法的对比**是我最大的收获。分类删除操作涉及"是否有菜品关联"和"是否有套餐关联"两个条件，场景法从用户操作出发设计"先建菜品→再删分类→应被拦截"的端到端流程，直观易懂；因果图法则通过列出所有"因"和"果"的逻辑关系，系统性地生成了 4 条用例——包括"同时关联菜品和套餐"这种场景法容易遗漏的复合场景。这让我认识到：**场景法适合集成测试阶段的"故事性"验证，因果图法适合单元测试阶段的"逻辑完备性"验证，两者不是二选一，而是互补**。

在技术层面，Redis 缓存的 Mock（`@MockBean RedisTemplate`）、H2 与 MySQL 的 SQL 语法差异处理、`DishFlavorMapper` XML 文件名含空格的调试，都让我积累了宝贵的实战经验。尤其是 H2 兼容性问题——单元测试中 SQL 通过不代表 MySQL 上也能正常运行——这让我深刻理解了"测试环境与生产环境的一致性"对测试有效性的影响。

### 9.4 M4 — 判定表法的系统化思维与文件上传测试的边界探索

M4 模块包含套餐管理和文件上传两大功能，共 50 个单元测试用例。这次实践有两个核心收获。

第一，**判定表法在多条件组合场景中的系统化优势**。以 `startOrStop`（套餐起售/停用）方法为例，该方法涉及 4 个条件的组合判断：目标状态（启用/停用）、菜品列表是否为 null、是否为空、是否包含未启售菜品。等价类划分法仅覆盖了 5 个等价类，而判定表法通过穷举条件组合生成了 7 条测试用例，发现了"菜品列表为 null 时正常启用"这一等价类容易遗漏的边界场景。这让我深刻理解了**判定表法的核心价值不在于"更多用例"，而在于"更系统的覆盖"——通过表格穷举，人为疏忽的概率大幅降低**。

第二，文件上传测试让我认识到**边界值不仅仅是数值的边界**。对于文件上传接口，边界值体现在文件大小（100B vs 5MB）、文件格式（JPG/PNG）、文件名为空字符串（`""` vs `null` 的微妙区别）、文件内容为空等多个维度。尤其是 `null` 和空字符串 `""` 的区别——在 Java 中它们是两种完全不同的状态，但开发者常混为一谈——这个看似细节的认知，在后续发现的 BUG-004（NPE）和 BUG-005（StringIndexOutOfBoundsException）中得到了验证。

### 9.5 M5 — 状态迁移图：让"隐形"的业务规则变得可见

M5 模块负责管理端订单操作和定时任务，是项目中**状态逻辑最密集**的模块。订单从待付款→待接单→已接单→派送中→已完成的 5 步状态链，每一步都存在"非法状态迁移"的边界。

**状态迁移图法**是我本次最大的方法性收获。在测试 Service 层时，我将订单的 6 种状态绘制为状态节点，将接单/拒单/取消/派送/完成/超时 6 个操作绘制为有向边，然后系统性地验证每条边的前置条件。这个方法直接产出了 9 个状态校验用例，覆盖了"已接单状态才能派送""派送中状态才能完成""已完成状态再派送被拦截"等边界。这种"先建模，再测试"的工作方式，让我意识到：**复杂系统的测试瓶颈往往不是"写多少用例"，而是"能否先看清业务规则的全貌"**。

定时任务的测试也让我积累了新的技术经验：通过直接调用 `OrderTask.processTimeoutOrder()` 而非等待 cron 触发，将测试执行时间从分钟级压缩到秒级；通过向 H2 数据库直接插入 20 分钟前的订单记录来模拟超时场景——这些"绕开框架"的测试技巧，在实际工程中非常实用。

### 9.6 M6 — 鉴权与隔离：从代码正确性到数据安全性的认知跃迁

M6 模块涵盖 C 端用户微信登录和地址簿管理，测试用例数虽不多（22 个），但在安全性和用户体验层面的思考深度远超预期。

**默认地址互斥切换**的业务逻辑是本次最精妙的设计。设置新默认地址时，Service 层需要先通过 `update` 将该用户所有现有地址的 `isDefault` 置为 0，再将目标地址的 `isDefault` 置为 1。这个"两步操作"在 Service 层单元测试中通过 `verify(AddressBookMapper, times(2)).update(any())` 进行验证。但这一逻辑的脆弱性在于：如果第一步成功、第二步失败（例如目标地址已被删除），将导致用户失去默认地址。这让我认识到：**单元测试断言 Mock 调用次数可以验证逻辑走通，但只有集成测试的完整数据回滚验证才能真正检验事务原子性**。

在鉴权层面，C 端 Token 无法访问管理端接口的隔离验证让我从"功能测试"升级到"权限测试"的视角。在大型项目中，越权访问往往不是功能性 Bug，而是安全性漏洞——这类缺陷一旦上线，修复成本远高于功能缺陷。这促使我将"权限边界验证"纳入个人今后的测试思维框架中。

### 9.7 M7 — 在业务流程最复杂的模块中学会"场景驱动测试"

M7 是苍穹外卖系统中**业务流程最长、代码逻辑最复杂**的模块，覆盖从 C 端浏览、加购、下单到支付的完整链路，共计 76 个单元测试用例和 5 个集成测试场景。这次实践让我对软件工程的两个核心问题有了深刻理解。

第一，**场景法在流程型业务中的不可替代性**。M7 的下单链路存在强时序依赖——必须先加购才能下单，必须先下单才能支付。正交实验法中"空购物车 × 有效地址"等因子组合在此语境下毫无意义。场景法通过模拟真实用户的操作序列（浏览→加购→改数量→清空→再加购→下单），不仅覆盖了代码路径，更验证了**业务流程本身的合理性**。这让我认识到：测试方法的选择不是"哪种更高级"，而是"哪种更适合被测对象的特征"。

第二，**数据隔离 Bug 的发现过程让我重新审视了单元测试的局限性**。`ShoppingCartServiceImpl.showShoppingCart()` 中 `.id(userId)` 应为 `.userId(userId)` 的 Bug，在单元测试的 Mock 环境中完全无法暴露（因为 Mock 不执行真实 SQL），在 H2 环境下也可能因表结构差异而侥幸通过。只有当集成测试用 Docker MySQL 真库执行时，才暴露出"所有用户看到相同购物车"的数据安全问题。这个案例给了我一个深刻的教训：**单元测试保证"逻辑正确"，集成测试保证"真实环境正确"，两者缺一不可**。

### 9.8 M8 — 从性能拐点到安全漏洞：系统测试的全景视角

作为 M8，我的职责覆盖了报表功能测试（24 个单元测试用例）、基础设施验证（`GlobalExceptionHandler`10 个异常分支 + `AutoFillAspect` 判定表法 4 条规则）、性能压力测试（JMeter 4 个并发级别）和安全测试（SQL 注入、XSS、越权访问、JWT 伪造）。

**JMeter 性能测试**让我第一次将"响应时间"和"TPS"从教科书概念转化为实践经验。50→200→500 并发梯度的测试结果清晰地展示了系统的性能拐点——TPS 在 200 并发达峰值 5,106 后在 500 并反而下降 6.9%。这个数据不仅产出了具体的优化建议（HikariCP 连接池扩容、Tomcat 线程池调整），更让我理解了**性能测试的本质不是"跑一个数字"，而是"找出瓶颈并定位根因"**。

**安全测试**则打开了一个全新的测试维度。SQL 注入测试中，在搜索框输入 `' OR '1'='1` 验证了 MyBatis 参数化查询的有效防护；越权测试中，用用户 A 的 Token 尝试访问用户 B 的订单验证了 JWT 鉴权的正确隔离。这些测试让我认识到：在大型项目中，安全性不是"附加功能"，而是需要在测试策略中从一开始就纳入的**基础质量属性**。

`GlobalExceptionHandler` 的 10 个异常分支测试让我养成了"异常路径和正常路径同等重要"的测试习惯。在真实生产环境中，异常路径的触发频率可能远低于正常路径，但一次未处理的异常就可能导致整个请求链路的崩溃——这种"低频高影响"的特征，决定了异常测试在质量保障中的特殊地位。

### 9.9 工程实践方法对大型项目开发的意义——团队共识

回顾为期 4 周的测试实践，8 名成员在各自模块中应用了等价类划分、边界值分析、判定表、因果图、状态迁移图、场景法、正交实验法共 **7 种测试用例设计方法**，搭建了 JUnit5 + Mockito + H2 + JMeter + Apifox + Docker + GitHub Actions + Jira 共 **8 项工具链**。这些工程实践方法对大型项目开发的意义，可以从以下三个维度总结。

**第一，方法论让质量保障从"经验驱动"走向"系统驱动"。** 在项目初期，我们对"如何设计测试用例"的认知停留在"想到什么测什么"的直觉层面。通过分别采用两种方法对同一功能设计测试用例并进行对比分析，我们深刻体会到：每一种测试设计方法都是一种"系统化的思维框架"——等价类保证代表性覆盖、判定表保证条件组合穷举、状态迁移保证状态边界完整。在大型项目中，代码规模指数级增长，仅靠经验直觉无法保证测试完备性，**方法论是应对复杂性的认知工具**。

**第二，自动化让质量保障具有可持续性。** 355 个测试用例如果全部手工执行，每次回归需要数小时，且容易遗漏。通过 GitHub Actions CI 实现每次 Push 自动运行全量测试，测试从"一次性活动"变成了"持续进行的过程"。在大型项目中，代码变更频繁、团队人员流动、发布周期压缩，**自动化测试是质量内建（Built-in Quality）的唯一可行路径**。

**第三，工程化协作让个人能力转化为团队资产。** Jira 的 Task 拆分与 Review 机制、Given-When-Then 的标准化描述、Docker 统一测试环境——这些工程实践将个人的测试工作转化为可审查、可追溯、可复用的团队知识。在大型项目中，个体英雄主义不可持续，**规范化流程和工具链是团队规模化协作的粘合剂**。

本次苍穹外卖测试实践让我们认识到：软件测试不是写代码的"附属工序"，而是一门有方法论支撑、有工具链保障、有工程管理规范的独立学科。掌握了这些方法，我们在未来面对更大规模、更复杂的软件系统时，将拥有从混沌中建立秩序的能力。
