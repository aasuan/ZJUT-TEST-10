### 遇到的问题与解决方案

#### 问题 1：JWT 拦截器阻断 MockMvc 请求

**现象**：Controller 测试使用 `@WebMvcTest` 启动后，所有 MockMvc 请求返回 401 未授权，无法到达 Controller。

**原因**：`JwtTokenAdminInterceptor` 在测试环境下仍会校验请求头中的 token，MockMvc 请求未携带有效 token 导致被拦截。

**解决方案**：在 `@BeforeEach` 中 Mock 拦截器的 `preHandle` 方法返回 `true`，绕过 JWT 鉴权：

```java
@MockBean
private JwtTokenAdminInterceptor jwtTokenAdminInterceptor;

@BeforeEach
void setUp() throws Exception {
    when(jwtTokenAdminInterceptor.preHandle(any(), any(), any())).thenReturn(true);
}
```

#### 问题 2：GlobalExceptionHandler 未被 WebMvcTest 加载

**现象**：Controller 层异常测试用例中，Mock 的 Service 抛出了 `BaseException`，但响应中未返回预期的错误 JSON 格式。

**原因**：`@WebMvcTest` 仅加载指定的 Controller，不会自动加载 `GlobalExceptionHandler`。

**解决方案**：在 `@WebMvcTest` 测试类上添加 `@Import(GlobalExceptionHandler.class)`，手动导入全局异常处理器。这使得异常测试用例能正确验证 `$.code` 为 0 和 `$.msg` 错误消息。

#### 问题 3：DishFlavorMapper XML 文件名含空格

**现象**：MyBatis 无法找到 `DishFlavorMapper` 的映射文件，Mapper 层测试启动报错。

**原因**：Mapper XML 文件名为 `DishFlavorMapper .xml`（`Mapper` 和 `.xml` 之间多了一个空格），MyBatis 自动映射机制无法识别。

**解决方案**：将文件重命名为 `DishFlavorMapper.xml`，与 Mapper 接口名完全一致。

#### 问题 4：MyBatis 测试依赖缺失

**现象**：编译阶段 `@MybatisTest` 注解无法解析，提示 `The import org.mybatis.spring.boot.test cannot be resolved`。

**原因**：`sky-server/pom.xml` 中缺少 `mybatis-spring-boot-starter-test` 依赖。

**解决方案**：在 `sky-server/pom.xml` 中添加 test scope 依赖，并在父 POM 中声明版本管理：

```xml
<dependency>
    <groupId>org.mybatis.spring.boot</groupId>
    <artifactId>mybatis-spring-boot-starter-test</artifactId>
    <version>3.0.2</version>
    <scope>test</scope>
</dependency>
```

#### 问题 5：PageHelper 分页在 Service 层 Mock 环境下不生效

**现象**：Service 层分页查询测试中，Mock 了 Mapper 返回的 Page 对象，但实际调用时 `PageHelper` 分页拦截器未生效。

**原因**：`MockitoExtension` 是纯 Mock 环境，不存在 Spring 容器，PageHelper 的 MyBatis 拦截器不会自动启动。

**解决方案**：Service 层测试中直接 Mock `Page` 对象，通过 `when(page.getTotal()).thenReturn(...)` 和 `when(page.getResult()).thenReturn(...)` 手动设置分页返回值，绕过 PageHelper 拦截器。Mapper 层测试中通过 `import PageHelperAutoConfiguration` 确保 PageHelper 正常工作。

#### 问题 6：H2 与 MySQL SQL 语法差异

**现象**：Mapper 层测试使用的 H2 内存数据库不支持部分 MySQL 特有语法（如 `AUTO_INCREMENT`、某些函数）。

**原因**：项目原 SQL 脚本针对 MySQL 编写，直接复用会导致建表失败。

**解决方案**：单独创建 `schema-h2.sql`，将 `AUTO_INCREMENT` 替换为 H2 兼容语法，将 `datetime` 替换为 `timestamp`，去除不支持的引擎指定语句。

### 遇到的问题与解决方案

#### 问题 1：Docker 环境连通性问题

**现象**：集成测试启动时无法连接 Docker MySQL，报 `CommunicationsException: Communications link failure`。

**原因**：Docker 容器未启动，或 `application-integration.yml` 中数据库连接配置与 `docker-compose.yml` 端口不一致。

**解决方案**：
1. 测试前先执行 `docker compose up -d` 在 `sky-take-out-main/` 目录下启动 MySQL + Redis
2. 确认 `application-integration.yml` 中 `spring.datasource.url` 端口为 3307，与 `docker-compose.yml` 中 MySQL 映射端口一致
3. 添加 `spring.sql.init.mode=never` 避免集成测试覆盖已有数据

#### 问题 2：集成测试数据冲突

**现象**：多个集成测试场景之间数据相互干扰，后执行的场景因前一个场景的数据残留而失败。

**原因**：测试场景涉及真实数据库操作，若不回滚则会相互影响。

**解决方案**：在 `DishIntegrationTest` 类上添加 `@Transactional` 注解，利用 Spring 事务管理在每个测试方法执行完毕后自动回滚。同时每个场景使用 `System.currentTimeMillis()` 生成唯一名称（如 `"集成测试分类_S1_" + timestamp`），避免同一事务内名称冲突。

#### 问题 3：集成测试中菜品 ID 获取困难

**现象**：`categoryService.save()` 和 `dishService.saveWithFlavor()` 方法返回值为 `void`，创建后无法直接获取 ID 进行后续验证。

**原因**：Controller 层接口通过 `Result.success()` 只返回成功状态，不返回创建的实体 ID。Service 层方法内部使用 `useGeneratedKeys` 回填 ID 到实体对象，但 Controller 不暴露此信息。

**解决方案**：通过分页查询接口按唯一名称反查刚创建的实体 ID：
```java
private Long getCategoryIdByName(String name) {
    CategoryPageQueryDTO query = new CategoryPageQueryDTO();
    query.setPage(1); query.setPageSize(10);
    query.setName(name);
    PageResult<Category> result = categoryService.pageQuery(query);
    return result.getRecords().isEmpty() ? null : result.getRecords().get(0).getId();
}
```
同时提取了 6 个辅助方法封装重复的反查逻辑和断言逻辑。
