## 3. 覆盖场景（对照分工）

### 场景 1：登录 → Token → 带 Token 访问业务接口 → 无 Token 401

- 使用 `IntegrationTestHelper.adminLogin` 登录获取 JWT
- 带 `token` 请求头访问：`GET /admin/employee/page`、`POST /admin/employee`、`PUT /admin/employee`
- 不带 Token 访问分页接口，断言 HTTP **401**

### 场景 2：错误密码登录

- `POST /admin/employee/login`，错误密码
- 断言 `code=0`，`msg=密码错误`（`GlobalExceptionHandler` 处理）

## 4. 测试方法对比（login）

对照《测试时间线》W3 要求，以 `POST /admin/employee/login`（`EmployeeServiceImpl.login`）为被测对象，分别用**等价类划分法**和**边界值分析法**设计用例，并在 W2 单元测试与 W3 集成测试中落地。

### 4.1 等价类划分法

**被测逻辑**：按用户名查库 → 校验密码 MD5 → 校验账号状态 → 返回员工信息 / 抛业务异常。

| 输入条件 | 有效等价类 | 无效等价类 |
|----------|-----------|-----------|
| 用户名 username | ① 库中存在且唯一匹配 | ④ 库中不存在 |
| 密码 password | ② 与库中 MD5 一致 | ⑤ 与库中 MD5 不一致 |
| 账号状态 status | ③ `ENABLE(1)` 可登录 | ⑥ `DISABLE(0)` 账号锁定 |

**等价类 → 用例映射（W2 Service / Controller）**：

| 类编号 | 覆盖等价类 | 测试方法 | 预期 |
|--------|-----------|----------|------|
| EC-1 | ①+②+③ | `login_success_returnsEmployee` / `login_success_returnsToken` | 登录成功，返回员工 / Token |
| EC-2 | ④ | `login_usernameNotFound_*` / `login_accountNotFound_*` | `账号不存在` |
| EC-3 | ①+⑤ | `login_wrongPassword_*` | `密码错误` |
| EC-4 | ①+②+⑥ | `login_accountDisabled_*` / `login_accountLocked_*` | `账号已锁定` |

等价类法共 **4 类业务结果**，W2 各层至少 1 条代表用例，保证每类输出都被覆盖一次。

### 4.2 边界值分析法

login 的输入为离散字符串，边界主要体现在**空值、临界正确值、与正确值相邻的错误值**：

| 边界项 | 边界取值 | 测试方法 | 说明 |
|--------|----------|----------|------|
| 用户名 | 不存在用户（`ghost` / `nouser`） | `login_usernameNotFound_*` | 有效域外侧：任意不存在名 |
| 密码 | 正确值 `123456` | `login_success_*` | 有效边界内代表值 |
| 密码 | 与正确值仅差一位（`wrong-password`） | `login_wrongPassword_*`、集成测 `login_wrongPassword_returnsBusinessErrorJson` | 相邻无效值，易触发真实校验逻辑 |
| 账号状态 | `ENABLE` vs `DISABLE` | `login_accountDisabled_*` | 状态枚举边界：唯一合法登录态为启用 |

边界值法侧重**贴近边界的代表输入**；本模块未对 `username=""` / `password=""` 单独建用例（当前接口未做参数校验，易返回 500，属已知缺口，W4 可补）。

### 4.3 两种方法比较

| 比较维度 | 等价类划分法 | 边界值分析法 |
|----------|-------------|-------------|
| 核心思想 | 按业务结果把输入空间划分为若干类，每类取一个代表值 | 在等价类边界及邻近点取极值、相邻值 |
| 用例数量 | 4 类结果 → 4 条核心用例 | 与等价类重叠，强调 password 相邻错误、status 0/1 |
| 完备性 | 保证四种业务分支不遗漏 | 补充「差一点就对」的密码场景，更易发现比较逻辑错误 |
| 对本接口适用度 | ⭐⭐⭐⭐⭐ login 分支清晰，四类结果一一对应 | ⭐⭐⭐⭐ 密码相邻错误、状态 0/1 边界有价值；空串边界依赖后续参数校验 |

**实践结论**：W2 以等价类为主框架完成 Service / Controller 全覆盖；边界值思想体现在错误密码、禁用状态等用例上。W3 集成测选取 **EC-3（密码错误）** 与 **EC-1 成功路径 + JWT** 做全链路复验，其余等价类已在 W2 验证，避免集成层重复堆砌。
