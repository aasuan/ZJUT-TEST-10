## 1. 测试目标

在 **真实 Spring 容器 + Docker MySQL/Redis** 环境下，验证管理端订单模块与定时任务相关组件的**端到端协作**，覆盖分工文档中 M5 集成测试三项场景：

| Jira | 场景 |
|------|------|
| SCRUM-261 | 下单 → 接单 → 派送 → 完成，完整状态链 |
| SCRUM-262 | 下单 → 拒单，订单状态与拒单原因入库 |
| SCRUM-263 | 模拟超时订单 → 触发 `OrderTask` → 自动取消 |

## 3. 测试设计说明

### 3.1 全链路场景（SCRUM-261 / 262）

采用 **C 端下单 + 支付 → 管理端操作** 的真实 HTTP 调用：

1. C 端：加购物车 → 提交订单 → 模拟支付（`payment` 内部走 `paySuccess`）  
2. 管理端：携带 `token` 请求头依次 **接单 / 派送 / 完成**（或 **拒单**）  
3. 使用 `OrderMapper.getById` 断言数据库中 `status` 与业务一致  

**鉴权**：

- 管理端：`IntegrationTestHelper.adminLogin` → 请求头 `token`  
- C 端：`IntegrationTestHelper.userToken(9001)` → 请求头 `authentication`（避免依赖微信 `jscode2session`）

**微信支付退款**：拒单场景若已支付会调用 `WeChatPayUtil.refund`，集成环境无证书，使用 `@MockBean WeChatPayUtil` 返回模拟成功，**不影响订单状态与拒单原因入库验证**。

### 3.2 定时任务场景（SCRUM-263）

向 `orders` 表插入 **20 分钟前、待付款** 的订单，直接调用 `OrderTask.processTimeoutOrder()`，断言：

- `status = 6`（已取消）  
- `cancel_reason = 超时未支付`  
- `cancel_time` 非空  

（不测 cron 调度，与 W2 单元测试分工一致。）

## 5. 测试方法对比（W3 要求）

M5 在 W2 已用 **状态迁移图 + 场景法** 设计用例，W3 对比如下：

| 维度 | 状态迁移图 | 场景法 |
|------|------------|--------|
| 适用点 | 非法状态、分支边界（拒单/派送/完成前置状态） | 端到端业务故事（下单到完成、拒单闭环） |
| 本模块对应用例 | IT-M5-03 超时取消；Service 单测中非法状态 | IT-M5-01、IT-M5-02 全链路 |
| 优点 | 覆盖状态边清晰、易发现状态机漏洞 | 贴近用户与商家操作路径 |
| 局限 | 需补充跨模块步骤才能成集成用例 | 对异常分支需额外设计 |

**结论**：集成测试以**场景法**为主组织用例（三个 Jira 子任务即三个场景）；**状态迁移图**用于补充超时取消、非法状态等边界，与单元测试形成互补。

## 7. 差异与风险

| 项 | 说明 |
|----|------|
| C 端登录 | 集成测试未走微信登录接口，使用 JWT 工具直接签发 Token |
| 退款 | `WeChatPayUtil` 使用 Mock，仅验证订单域逻辑 |
| 数据隔离 | `@Transactional` + `integration-m5-setup.sql` 清理/重建 9001 号测试数据 |
| Docker 依赖 | 未启动 Docker 时集成测试无法连接 3307/6380，会启动失败 |
