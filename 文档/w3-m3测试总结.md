### 2.2 集成测试场景

#### 场景一：分类下存在菜品时删除分类应抛出异常

| 步骤 | 操作 | 验证点 |
|------|------|--------|
| 1 | 通过 `categoryService.save()` 创建新分类 | 分页查询可检索到该分类 |
| 2 | 通过 `dishService.saveWithFlavor()` 创建菜品关联该分类 | 分页查询可检索到该菜品，categoryId匹配 |

| 4 | 再次分页查询该分类 | 分类依然存在于数据库中 |

**验证的核心规则**：分类-菜品强关联删除拦截，全链路验证 Service → Mapper → DB。

#### 场景二：菜品口味生命周期 —— 增/改/删后联表更新验证

| 步骤 | 操作 | 验证点 |

| 2 | 修改口味：辣度→中辣（改）、保留甜度、新增酸度/微酸、删除咸度 | 共3条口味（删1增1改1，总数不变） |
| 3 | 再次查询菜品详情 | 辣度变为中辣、甜度不变、酸度存在、咸度已删除 |

**验证的核心规则**：`updateWithFlavor()` 的三步操作（删旧→更新菜品→插新）在真实数据库联表场景下的正确性。

#### 场景三：批量删除 —— 起售菜品阻断删除，停售菜品物理删除

| 步骤 | 操作 | 验证点 |
|------|------|--------|
| 1 | 创建分类 + 创建3个停售菜品 | 3个菜品均入库，ID不重复 |
| 2 | 将菜品1起售（status=ENABLE） | 状态变更生效 |
| 3 | 批量删除全部3个菜品（含1起售） | 抛出 `DeletionNotAllowedException`（消息 `DISH_ON_SALE`） |
| 4 | 验证删除失败后3个菜品均仍存在 | 事务回滚，数据未受影响 |
| 5 | 仅删除2个停售菜品 | 数据库减少2条，起售菜品仍存在 |
| 6 | 验证停售菜品的口味数据一并清除 | `dishFlavorMapper.getByDishId()` 返回空列表 |

**验证的核心规则**：
- `deleteBatch()` 中起售菜品阻断全量删除（任一菜品起售则整批拒绝）

## 3. 测试方法对比：场景法 vs 因果图

按照 W3 测试任务分工，M3 模块选取 **"菜品-分类关联"** 业务场景，分别采用场景法和因果图法设计测试用例并进行对比分析。

### 3.1 场景法设计用例

场景法通过模拟用户实际操作流程，将业务操作串联为端到端场景。

**场景一：分类-菜品关联删除拦截**

```
业务流程：管理员创建"湘菜"分类 → 创建"剁椒鱼头"菜品关联该分类 → 尝试删除"湘菜"分类 → 系统拒绝并提示"当前分类关联了菜品,不能删除" → 分类数据仍存在
```

| 场景编号 | 场景描述 | 事件流 | 预期结果 |
|----------|----------|--------|----------|
| SC-M3-01 | 分类有关联菜品时删除被拦截 | S1: 创建分类 → S2: 创建菜品关联该分类 → S3: 删除分类 | 步骤S3抛出DeletionNotAllowedException，分类仍存在 |

**场景二：菜品口味生命周期管理**

```
业务流程：创建"麻辣小龙虾"菜品（含辣度/麻度/口味3条口味）→ 查看菜品详情确认3条口味 → 修改口味（辣度改为"重辣"、新增"分量"、删除"麻度"）→ 再次查看确认口味变更正确
```

| 场景编号 | 场景描述 | 事件流 | 预期结果 |
|----------|----------|--------|----------|
| SC-M3-02 | 菜品口味增删改全生命周期 | S1: 创建含3条口味的菜品 → S2: 查询验证3条口味 → S3: 修改口味（改1/增1/删1）→ S4: 查询验证修改生效 | 口味变更正确：修改生效、新增存在、已删除的不存在 |

**场景三：菜品批量删除状态校验**

```
业务流程：创建3个菜品（A、B、C均为停售）→ 将A起售 → 尝试批量删除A、B、C → 系统拒绝 → 仅删除B、C → 验证B、C已物理删除、A仍存在
```

| 场景编号 | 场景描述 | 事件流 | 预期结果 |
|----------|----------|--------|----------|
| SC-M3-03 | 批量删除时起售菜品阻断 | S1: 创建3个停售菜品 → S2: 起售其中1个 → S3: 批量删除全部3个 → S4: 仅删除2个停售菜品 | S3抛异常，S4成功后仅剩1个起售菜品 |

**场景法共设计 3 条用例**，覆盖了分类-菜品关联删除拦截、口味生命周期、批量删除状态校验。

### 3.2 因果图法设计用例

因果图法通过分析"原因"（输入条件）和"结果"（预期输出）之间的逻辑关系，系统性地生成测试用例。

**被测功能**：分类删除操作 —— 删除分类时需检查是否有关联的菜品或套餐。

**原因列表**：

| 原因ID | 描述 |
|--------|------|
| C1 | 分类下有关联菜品（dishCount > 0） |
| C2 | 分类下有关联套餐（setmealCount > 0） |
| C3 | 分类下无任何关联数据 |

**结果列表**：

| 结果ID | 描述 |
|--------|------|
| E1 | 删除成功，分类从数据库移除 |
| E2 | 抛出 DeletionNotAllowedException（消息：CATEGORY_BE_RELATED_BY_DISH） |
| E3 | 抛出 DeletionNotAllowedException（消息：CATEGORY_BE_RELATED_BY_SETMEAL） |

**因果图逻辑关系**：

- C1 → E2（有关联菜品则拒绝删除）
- C2 → E3（有关联套餐则拒绝删除）
- C3 = NOT(C1 OR C2) → E1（无任何关联则允许删除）

**因果图生成的测试用例**：

| 用例编号 | 原因组合 | 覆盖的逻辑 | 预期结果 | 对应测试 |
|----------|----------|-----------|----------|----------|
| CE-M3-01 | C3（无关联） | NOT(C1 OR C2) → E1 | 删除成功 | CategoryServiceImplTest.TC-CAT-201 |
| CE-M3-02 | C1（有关联菜品） | C1 → E2 | 抛异常，消息含DISH | CategoryServiceImplTest.TC-CAT-202 |
| CE-M3-03 | C2（有关联套餐） | C2 → E3 | 抛异常，消息含SETMEAL | CategoryServiceImplTest.TC-CAT-202补充 |
| CE-M3-04 | C1 AND C2（同时关联菜品和套餐） | C1 → E2（C1优先级高于C2，按源码判断顺序） | 抛异常，消息含DISH | 集成测试场景一（全链路验证） |

**因果图法共设计 4 条用例**，覆盖了删除操作的3条因果路径及1条复合条件路径。

### 3.3 两种方法的比较分析

| 比较维度 | 场景法 | 因果图法 |
|----------|--------|----------|
| **核心思想** | 模拟用户实际操作流程，将多个操作串联为业务场景，从端到端角度验证系统行为 | 分析输入条件（因）与输出结果（果）之间的逻辑关系，通过因果图系统生成覆盖所有逻辑组合的用例 |
| **适用场景** | 业务流程长、操作步骤多、需要验证多步骤组合正确性的场景 | 条件组合逻辑复杂、输入与输出之间存在明确的AND/OR/NOT关系的场景 |
| **用例数量** | 3条 | 4条 |
| **用例完备性** | 中等。场景法依赖设计者经验选取典型场景，可能遗漏非典型的条件组合路径 | 较高。因果图通过逻辑分析穷举所有条件组合，系统性更强 |
| **可读性** | 高。场景用例贴近用户实际操作，非技术人员也可理解 | 中等。因果图需要先理解条件-结果映射关系，对非技术人员不够直观 |
| **缺陷发现能力** | 中等偏高。擅长发现业务流程层面的缺陷（如多步骤操作后状态不一致） | 较高。擅长发现条件判断遗漏（如缺少某个条件分支导致误删/漏拦） |
| **对本被测功能的适用度** | ⭐⭐⭐⭐ 业务流程特征明显（创建→关联→删除），场景法能直观验证"分类-菜品关联"的端到端行为 | ⭐⭐⭐⭐⭐ 删除分类涉及多个条件判断（菜品关联、套餐关联），因果图能系统覆盖所有组合路径，不会遗漏 |
| **维护成本** | 较低。场景描述简洁，业务变更时易于调整 | 中等。条件增减时需重新分析因果逻辑关系并调整用例 |

### 3.4 方法选择的综合建议

**场景法**更适合 M3 模块的**集成测试阶段**。菜品-分类关联的业务场景具有明显的流程特征：创建分类→创建菜品→尝试删除→验证拦截，这种端到端的操作链天然适合用场景法描述。集成测试中的3个场景均采用场景法思路设计，直观且完整地验证了核心业务链路。

**因果图法**更适合 M3 模块的**单元测试阶段**，尤其是 Service 层 `deleteById` 方法的测试设计。该方法的条件判断逻辑明确（dishCount > 0 → 抛异常；setmealCount > 0 → 抛异常；均为0 → 删除），用因果图法可以系统性保证所有条件组合被覆盖，不会遗漏"同时存在菜品和套餐关联"这样的复合场景。

**本次实践结论**：两种方法互补使用效果最佳。W2 单元测试阶段，Service 层 `deleteById` 的 3 条测试用例采用了因果图法的思路，确保了条件组合的完备覆盖；W3 集成测试阶段，`DishIntegrationTest` 的 3 个场景采用了场景法思路，从端到端角度验证了全链路的正确性。两者结合，既保证了逻辑覆盖的完备性，又保证了业务流程验证的真实性。

## 4. W3 测试过程中的差异与问题

### 4.1 环境配置差异

1. **Docker 环境依赖**：集成测试需要 Docker Compose 启动 MySQL（端口3307）+ Redis（端口6380），测试前需确保 `docker compose up -d` 已执行。与 W2 单元测试使用 H2 内存数据库不同，W3 集成测试使用真实数据库验证 SQL 兼容性和全链路行为。

2. **Profile 切换**：W2 单元测试中 Mapper 层使用 `application-test.yml`（H2），Controller/Service 层无需数据库；W3 集成测试统一使用 `application-integration.yml`（Docker MySQL），通过 `@ActiveProfiles("integration")` 切换。

3. **SQL 初始化策略**：集成测试中设置 `spring.sql.init.mode=never`，避免每次启动覆盖 Docker 数据库中的已有数据。

### 4.2 测试执行差异

1. **事务回滚机制**：`DishIntegrationTest` 使用 `@Transactional` 确保每个测试场景执行完毕后自动回滚，不污染共享数据库。这与 W2 Mapper 层测试的 `@Transactional` 回滚策略一致。

2. **辅助方法提取**：集成测试类中提取了 `getCategoryIdByName()`、`getDishIdByCategory()`、`createDishAndGetId()`、`countDishesInCategory()`、`assertFlavorExists()`、`assertFlavorNotExists()` 共 6 个辅助方法，提高了测试代码的可读性和复用性。

### 4.3 未发现缺陷

W3 集成测试阶段**未发现新的缺陷**，3 个场景全部一次通过。分析原因：

1. W2 单元测试已充分覆盖各层的正常路径和异常路径，代码质量基线较高
2. 分类-菜品关联的业务逻辑相对成熟，条件判断清晰
3. 集成测试场景与 W2 单元测试用例形成互补，验证了各层组合后的端到端正确性

2. **菜品-口味联表操作**：`DishServiceImpl.saveWithFlavor()` 和 `updateWithFlavor()` 采用"先插菜品→再插口味"和"删旧口味→更新菜品→插新口味"的模式，通过集成测试场景二验证了联表操作在真实数据库中的正确性。

3. **批量删除事务一致性**：`DishServiceImpl.deleteBatch()` 采用"全量校验→全量操作"的模式，任一菜品不满足删除条件则整批拒绝。集成测试场景三验证了事务回滚后数据完整性。

4. **Redis 缓存清除**：DishController 测试验证了新增/修改/删除/启停菜品时均触发 `redisTemplate.keys("dish_*")` 清除缓存，保证了缓存一致性。
