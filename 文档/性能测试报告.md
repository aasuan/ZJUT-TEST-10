### 2.4 Token 获取机制

#### 用户端 Token 流程

1. **发送登录请求：** POST `/user/user/login`，携带 `code=testcode`
2. **Mock 逻辑触发：** 服务端识别 `testcode`，返回固定用户信息
3. **生成 JWT：** 系统根据用户 ID=1 生成 JWT Token
4. **提取 Token：** JMeter 使用 JSON PostProcessor 提取 `$.data.token`
5. **保存变量：** 将 Token 保存为 `user_token` 变量
6. **后续使用：** 在后续请求的请求头中使用 `${user_token}`

#### 管理端 Token 流程

1. **发送登录请求：** POST `/admin/employee/login`，携带用户名密码
2. **服务端验证：** 验证 admin/123456
3. **生成 JWT：** 系统根据员工信息生成 JWT Token
4. **提取 Token：** JMeter 使用 JSON PostProcessor 提取 `$.data.token`
5. **保存变量：** 将 Token 保存为 `admin_token` 变量
6. **后续使用：** 在管理端请求的请求头中使用 `${admin_token}`

#### HTTP 信息头管理器配置

每个需要认证的请求都配置了 Header Manager：

```
Authorization: Bearer ${user_token}  （用户端接口）
Authorization: Bearer ${admin_token}  （管理端接口）
Content-Type: application/json
```

### 4.3 性能趋势分析

#### TPS 变化趋势

- **50 → 100 并发：** TPS 上升 2.5%（4864 → 4986），响应时间增加 95%（9.38ms → 18.32ms）
- **100 → 200 并发：** TPS 上升 2.4%（4986 → 5106），响应时间增加 96%（18.32ms → 35.89ms）
- **200 → 500 并发：** TPS 下降 6.9%（5106 → 4755），响应时间增加 169%（35.89ms → 96.41ms）

**分析：** 
- TPS 在 200 并发时达到峰值 5106 requests/s，之后在 500 并发时略有下降
- 响应时间随并发增加呈稳定增长趋势，从 9.38ms 增长到 96.41ms，增长了约 10 倍
- 系统在 50-200 并发范围内 TPS 持续增长，表现优秀
- 500 并发时 TPS 开始下降，响应时间显著增加，表明系统接近承载极限

---

#### 响应时间变化趋势

- **50 并发：** 平均 9.38 ms，90% 43.0 ms
- **100 并发：** 平均 18.32 ms，90% 82.0 ms
- **200 并发：** 平均 35.89 ms，90% 161.0 ms
- **500 并发：** 平均 96.41 ms，90% 250.0 ms

**分析：** 
- 响应时间随并发数增加而稳定增长，500 并发时平均响应时间 < 100ms，仍在可接受范围
- 50 并发时响应时间极佳（9.38ms），用户体验非常流畅
- 90% 响应时间在 500 并发时达到 250ms，表明少数请求存在延迟
- 最大响应时间在 500 并发时达到 4045ms（约 4 秒），存在极端慢请求，需关注

### 4.4 性能瓶颈识别

#### 拐点分析

- **TPS 拐点：** 在 200 并发时，TPS 达到峰值 5106 requests/s，500 并发时下降到 4755 requests/s（下降 6.9%）
- **响应时间拐点：** 在 200→500 并发区间，响应时间从 35.89ms 增加到 96.41ms，增长速率显著加快；90%响应时间从 161ms 增加到 250ms
- **最大响应时间异常：** 500 并发时最大响应时间达到 4045ms，远超其他级别，表明存在严重的资源竞争或锁等待

#### 瓶颈定位

根据测试结果，系统性能瓶颈可能出现在：

1. **数据库层：** 
   - 高并发下数据库连接池可能成为瓶颈
   - 订单提交涉及多表操作（orders + order_detail），高并发下可能存在事务锁等待
   - 500 并发时最大响应时间达到 4045ms，暗示存在严重的慢查询或锁竞争
   - 订单条件搜索接口在 500 并发时平均响应时间达到 253ms，可能是复杂查询导致

2. **应用层：** 
   - Tomcat 线程池在 500 并发时可能接近上限
   - JWT Token 生成和验证在高并发下的累积开销
   - Mock 微信登录虽然简化了流程，但 500 并发时用户端登录平均响应时间仍达到 57ms

3. **网络层：** 
   - 本地测试网络延迟极低，但 500 并发时仍出现极端响应时间，说明问题主要在应用和数据库层
   - HTTP Keep-Alive 连接复用效果良好，但不是主要瓶颈

## 五、测试结论与建议

### 5.1 测试结论

#### 系统承载能力

- **最大稳定并发数：** 500 并发（测试上限，实际可能更高）
- **最大 TPS：** 5303 requests/s（50 并发时达到）
- **可接受的响应时间：** 在 500 并发下，平均响应时间 95ms < 100ms，符合预期

#### 系统稳定性

- **错误率：** 在所有测试级别下，错误率为 0.00%，系统稳定性优秀
- **资源使用：** 本地测试环境下，CPU 和内存使用率在正常范围内，未出现资源耗尽

---

### 5.2 优化建议

#### 短期优化（立即实施）

1. **优化 Ramp-Up 时间**
   - 问题描述：当前 Ramp-Up = 10秒，对于 500 并发来说过短，导致瞬时冲击
   - 优化方案：调整为 Ramp-Up = 50秒，使线程启动更平缓
   - 预期效果：获得更真实的性能数据，避免瞬时压力导致的异常

2. **添加思考时间（Think Time）**
   - 问题描述：当前测试无思考时间，所有请求连续发送，不符合真实场景
   - 优化方案：在关键操作间添加 1-3 秒随机延迟
   - 预期效果：模拟真实用户行为，降低服务器压力，获得更准确的评估

3. **分离用户端和管理端测试**
   - 问题描述：当前测试混合了用户端和管理端接口，流量比例不符合实际
   - 优化方案：创建独立的测试计划，分别评估两端性能
   - 预期效果：更精准地定位各端的性能瓶颈

---

#### 长期优化（规划实施）

1. **数据库优化**
   - 添加索引：确保 orders、order_detail、dish 等表的常用查询字段已建立索引
   - 读写分离：当并发超过 1000 时，考虑引入主从复制，读操作走从库
   - 缓存策略：引入 Redis 缓存菜品列表、套餐列表等热点数据，减少数据库查询

2. **应用层优化**
   - 线程池调优：调整 Tomcat 线程池配置（maxThreads、minSpareThreads）
   - JVM 参数优化：根据堆内存使用情况调整 -Xms、-Xmx，优化 GC 策略
   - 异步处理：订单提交后的非关键操作（如发送通知）改为异步执行

3. **架构优化**
   - 负载均衡：生产环境部署 Nginx 实现负载均衡，支持水平扩展
   - 微服务拆分：将用户服务、订单服务、菜品服务拆分为独立微服务
   - CDN 加速：静态资源（图片、CSS、JS）使用 CDN 分发

---

### 5.3 测试局限性

1. **测试场景简化：** 仅测试了 7 个核心接口，未覆盖全部业务场景（如支付、退款、评价等）
2. **数据量有限：** 测试数据较少（仅几条记录），未能模拟真实生产环境的万级数据量
3. **单一节点：** 仅在本地单节点测试，未考虑分布式部署和多实例场景
4. **Ramp-Up 较短：** Ramp-Up 时间为 10 秒，可能导致瞬时冲击，建议调整为更平缓的梯度
5. **缺少思考时间：** 未模拟真实用户的操作间隔，压力偏大，实际生产环境压力会更小
6. **Mock 登录：** 使用了 Mock 微信登录，真实环境的微信 API 调用会增加响应时间
7. **测试时长较短：** 每个测试仅运行 60 秒，未进行长时间稳定性测试（如 30 分钟以上）
