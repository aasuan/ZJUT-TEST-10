# M7 W3 集成测试总结

## 1. 概述

- **模块**：C端购物车 + 下单 + 支付（ShoppingCartController / OrderController(user) / PayNotifyController）
- **测试类型**：集成测试（W3）
- **测试框架**：JUnit5 + Spring Boot Test + MockMvc + Docker MySQL/Redis
- **基类**：`BaseIntegrationTest`（`@SpringBootTest` + `@AutoConfigureMockMvc` + `@ActiveProfiles("integration")`）
- **前置条件**：`docker compose up -d` 启动 MySQL(3307) + Redis(6380)

## 3. 测试方法对比

M7模块的测试用例设计方法对比：**场景法 vs 正交实验法**

### 3.1 场景法设计（本集成测试采用）

场景法从用户操作流出发，模拟真实业务流程：

| 维度 | 评价 |
|---|---|
| 优点 | 贴近真实用户行为、覆盖端到端流程、易于理解沟通 |
| 缺点 | 场景数量受限、边界条件可能遗漏 |
| 适用场景 | 流程复杂的业务链路（购物车→下单→支付） |

**本模块用场景法设计的5个场景：**

1. **基础流**：浏览→加购→改数量→清空→再购→下单（覆盖购物车全部CRUD+订单提交）
2. **支付流**：下单→支付→状态变更（覆盖支付状态机）
3. **隔离流**：双用户并发操作购物车（覆盖数据隔离）
4. **幂等流**：重复提交订单（覆盖空购物车拦截）
5. **异常流**：空购物车提交→回滚→正常提交（覆盖事务原子性）

### 3.2 正交实验法对比

如用正交实验法设计，需识别因子和水平：

| 因子 | 水平 |
|---|---|
| 购物车状态 | 空 / 有商品1件 / 有商品多件 |
| 地址 | 有效 / 无效(不存在) |
| 支付方式 | 微信(1) / 支付宝(2) |
| 菜品类型 | 单品 / 套餐 |
| 口味 | 有 / 无 |

正交表 L8(2^7) 可产生8个用例，覆盖两两组合。但正交实验法更适用于**输入参数组合多**的场景（如搜索筛选），对**流程型**业务（购物车→下单→支付）不如场景法直观。

### 3.3 方法对比结论

| 对比维度 | 场景法 | 正交实验法 |
|---|---|---|
| M7模块适用性 | **更优** — 业务流程是线性的 | 偏弱 — 因子间有先后依赖关系 |
| 用例数量 | 5个场景覆盖核心路径 | L8需8个用例 |
| 可读性 | 高 — 每个场景即一个故事 | 中 — 需映射回业务 |
| 边界覆盖 | 依赖设计者经验 | 系统化覆盖因子组合 |
| 推荐场景 | 流程型业务（下单、支付） | 参数组合型（搜索、筛选） |

**结论：M7模块更适合场景法。** 下单链路各步骤存在强时序依赖（必须先加购才能下单，必须先下单才能支付），正交实验法中"空购物车×有效地址"等组合无实际意义。

## 7. 总结

- **5个集成测试场景全部实现**，覆盖C端购物车CRUD、订单提交、支付状态变更、用户数据隔离、幂等性拦截、事务原子性
- **发现并修复1个Bug**：购物车查询未按用户隔离（数据安全缺陷）
- **测试方法对比**：场景法比正交实验法更适合流程型业务（购物车→下单→支付链路）
- **每个测试类独立可运行**，继承 `BaseIntegrationTest`，自带 `@BeforeEach`/`@AfterEach` 数据准备与清理
