### BUG-001：购物车查询未按用户隔离

| 属性 | 内容 |
|---|---|
| **缺陷编号** | BUG-001 |
| **严重级别** | 高（High） |
| **所属模块** | M7 — C端购物车 + 下单 + 支付 |
| **发现阶段** | 集成测试（W3） |
| **发现人** | M7 |
| **发现日期** | W3（5月19日—25日） |
| **缺陷位置** | `ShoppingCartServiceImpl.java` 第85行 `showShoppingCart()` 方法 |
| **缺陷描述** | 构建查询条件时使用 `.id(userId)` 代替 `.userId(userId)`，导致购物车列表查询未按用户过滤，**所有用户看到相同的购物车数据** |

| **经验教训** | 数据隔离类测试用例应尽早覆盖；字段名差异在单元测试 H2 环境下不易暴露，集成测试用真实 MySQL 验证更有效 |

- `W3-M7-集成测试总结.md`（第47行、第124-130行）

### BUG-002：MyBatis 参数绑定错误导致订单提交失败

| 属性 | 内容 |
|---|---|
| **缺陷编号** | BUG-002 |
| **严重级别** | 致命（Blocker） |
| **所属模块** | M7 — C端购物车 + 下单 + 支付 |
| **发现阶段** | 系统测试（W4） |
| **发现人** | 待补充 |
| **发现日期** | W4（5月26日—6月1日） |
| **缺陷位置** | `sky-server/src/main/resources/mapper/OrderDetailMapper.xml` 第8行 `<foreach>` 标签 |
| **缺陷描述** | `<foreach>` 标签使用 `collection="orderDetailList"`，但 Mapper 接口方法参数是单个 `List` 类型且未使用 `@Param` 注解，MyBatis 无法识别该参数名。运行时报 `BindingException: Parameter 'orderDetailList' not found. Available parameters are [arg0, collection, list]` |
| **根本原因** | MyBatis 参数绑定规范问题：当 Mapper 接口方法参数为单个 `List` 类型且未用 `@Param` 显式命名时，MyBatis 默认参数名为 `list`/`collection`，而非方法签名中的形参名。`<foreach collection="orderDetailList">` 引用了不存在的命名参数 |
| **影响范围** | **用户下单功能完全不可用**。所有 C 端用户提交订单时均会抛出 `BindingException`，订单无法入库，属于阻塞性缺陷 |
| **修复方案** | 将 `<foreach collection="orderDetailList">` 改为 `<foreach collection="list">`（或为 Mapper 接口参数添加 `@Param("orderDetailList")` 注解） |
| **回归测试** | 修复后需对下单全链路进行回归验证（购物车→下单→订单入库→明细入库） |
| **经验教训** | MyBatis Mapper XML 中的 `<foreach>` 集合名必须与 MyBatis 识别的参数名一致（单参数 List 默认名 `list`，或使用 `@Param` 显式指定）。此类错误应在集成测试阶段覆盖，单元测试中 Mock 了 Mapper 层可能遗漏。同时建议引入 MyBatis 单元测试（`@MybatisTest`）覆盖批量插入场景，可提前暴露参数绑定问题 |

**错误日志**：
```
org.apache.ibatis.binding.BindingException: Parameter 'orderDetailList' not found. 
Available parameters are [arg0, collection, list]
```

### BUG-003：营业额统计函数日期循环逻辑错误 + 缺少日期合法性校验

| 属性 | 内容 |
|---|---|
| **缺陷编号** | BUG-003 |
| **严重级别** | 高（High） |
| **所属模块** | M8 — 报表 + 基础设施 |
| **发现阶段** | 系统测试（W4） |
| **发现人** | 待补充 |
| **发现日期** | W4（5月26日—6月1日） |
| **缺陷位置** | `com.sky.service.impl.ReportServiceImpl.java` — `getTurnoverReport(LocalDate begin, LocalDate end)` 方法 |
| **缺陷描述** | 该函数存在两个逻辑错误：**① 缺少 begin 开始日期是否早于 end 结束日期的合法性校验**，若 begin 晚于 end 则统计区间无效，函数仍会执行并返回无意义数据；**② 日期循环条件写反**：原代码循环条件为 `begin.equals(end)`（begin 等于 end 才进入循环），正确逻辑应为 `!begin.equals(end)`（begin 不等于 end 时持续迭代），导致循环只在 begin == end 时执行一次，统计区间内除首日外的其余日期均被跳过 |
| **根本原因** | ① 缺少输入参数前置校验；② 循环条件布尔逻辑写反（`==` 误写为 `!=` 或其等价形式），属于典型的逻辑运算符使用错误 |
| **影响范围** | **营业额统计报表功能异常**。多日统计场景下仅返回首日数据，其余日期营业额缺失；且即使传入非法日期区间（begin > end）也不会报错，输出数据不可信 |
| **修复方案** | ① 方法开头增加 `if (begin.isAfter(end))` 判断，抛出 `IllegalArgumentException` 或返回业务异常；② 将循环条件修正为 `!begin.equals(end)`（或 `begin.isBefore(end)`），确保遍历 begin 到 end 的全部日期 |
| **回归测试** | 修复后需对单日/多日/跨月/非法区间四种场景进行回归验证，确保每日营业额数据完整且非法区间正确拦截 |
| **经验教训** | 日期区间类函数的测试必须覆盖三个等价类：合法区间（begin < end）、单日区间（begin == end）、非法区间（begin > end）。循环条件的布尔逻辑应优先使用语义明确的方法（如 `isBefore()`），避免 `==`/`!=` 的直觉性误写 |

### BUG-004：文件上传接口未对 null 文件判空导致 NullPointerException

| 属性 | 内容 |
|---|---|
| **缺陷编号** | BUG-004 |
| **严重级别** | 中（Medium） |
| **所属模块** | M4 — 套餐管理 + 文件上传 |
| **发现阶段** | 系统测试（W4） |
| **发现人** | 待补充 |
| **发现日期** | W4（5月26日—6月1日） |
| **缺陷位置** | `CommonController.java` — `upload(MultipartFile file)` 方法 |
| **缺陷描述** | 当前端/测试请求中未携带 `file` 参数时，`file` 对象为 `null`，代码直接调用 `file.getOriginalFilename()` 触发 `NullPointerException`，接口返回 HTTP 500 而非友好的业务错误信息 |
| **根本原因** | 缺少对 `MultipartFile` 参数的 `null` 值防御性检查 |
| **影响范围** | 无文件参数的请求会导致服务器 500 错误，而非返回 `"文件不能为空"` 的业务提示。影响 API 调用的友好性和健壮性 |
| **修复方案** | 在方法开头添加 `if (file == null) { return Result.error("文件不能为空"); }` 判空处理 |
| **回归测试** | 修复后 `testUpload_NoFile` 应返回 `code=0, msg="文件不能为空"` 而非 500 |
| **经验教训** | Controller 层对 `MultipartFile`、`@RequestBody` 等框架注入参数也应做 `null` 防御，不能假设框架一定拦截。单元测试中 `@WebMvcTest` + Mock 环境可能不会触发真实 NPE，需在集成/系统测试阶段补充无参数场景验证 |

### BUG-005：文件上传空文件名导致 StringIndexOutOfBoundsException

| 属性 | 内容 |
|---|---|
| **缺陷编号** | BUG-005 |
| **严重级别** | 中（Medium） |
| **所属模块** | M4 — 套餐管理 + 文件上传 |
| **发现阶段** | 系统测试（W4） |
| **发现人** | 待补充 |
| **发现日期** | W4（5月26日—6月1日） |
| **缺陷位置** | `CommonController.java` — `upload(MultipartFile file)` 方法中文件名处理逻辑 |
| **缺陷描述** | 当上传文件名为空字符串 `""` 时，`originalFilename != null` 条件为 `true`（空字符串不是 null），但 `"".lastIndexOf(".")` 返回 `-1`，随后 `substring(-1)` 抛出 `StringIndexOutOfBoundsException`，接口返回 500 错误 |
| **根本原因** | 空字符串 `""` 与 `null` 是不同的语义，代码仅做了 `!= null` 判断，未覆盖空字符串的边界情况。`lastIndexOf(".")` 在找不到分隔符时返回 `-1`，直接传给 `substring()` 会导致越界 |
| **影响范围** | 上传空文件名的请求会导致服务器 500 错误，属于健壮性缺陷 |
| **修复方案** | 将判断条件从 `if (originalFilename != null)` 改为 `if (originalFilename != null && !originalFilename.isEmpty())`，确保空字符串走默认后缀 `.jpg` 的容错逻辑 |
| **回归测试** | 修复后 `testUpload_EmptyFilename` 应正常返回 `code=1` 并使用默认 `.jpg` 后缀 |
| **经验教训** | 字符串判空应使用 `StringUtils.hasText()` 或同时检查 `!= null` 和 `!isEmpty()`，避免混淆 Java 中 `null` 和空字符串 `""` 两种不同的"无值"状态。文件名的扩展名提取前应先确认 `lastIndexOf(".") >= 0` |

## 七、缺陷分析小结

1. **缺陷总数为 5 个**：355 个单元测试 + 32 个集成测试场景中最终发现 5 个正式 Bug，整体代码质量基线较好。

2. **80% 的 Bug 来自系统测试阶段（W4）**：BUG-002~005 均在端到端系统测试中暴露，包括 MyBatis 参数绑定错误、日期循环逻辑错误、NPE 防御缺失等。这类缺陷在单元测试的 Mock 环境和集成测试的有限场景中均不易暴露，**验证了系统测试不可替代的价值**。

3. **BUG-002 为致命级阻塞缺陷**：MyBatis `<foreach>` 参数绑定错误导致用户下单功能完全不可用，属于典型的"编译通过但运行时报错"的配置类缺陷。建议在 Mapper 层单元测试中增加批量插入场景覆盖，可提前暴露此类参数绑定问题。

4. **BUG-003 暴露逻辑运算符误写问题**：日期循环条件 `==` 与 `!=` 的混淆是典型的人为失误，同时缺少日期合法性校验也反映了防御性编程意识不足。

5. **BUG-004/BUG-005 均为防御性编程缺失**：`null` 与空字符串的判空遗漏、`lastIndexOf` 返回值未校验，均为 Java 开发中的常见疏忽。建议在 Controller 层统一增加 `@Validated` + JSR-303 校验注解。

6. **改进点集中在边界/健壮性处理**：M4 的文件类型未限制、参数校验缺失，以及 M5 的 confirm 无状态校验，都属于"功能可用但不够健壮"的边界问题。

3. **改进点集中在边界处理**：M4 的文件类型未限制、参数校验缺失，以及 M5 的 confirm 无状态校验，都属于"功能可用但不够健壮"的边界问题，不阻塞核心业务流程。

4. **环境问题数量较多但均属正常**：17 个环境/配置问题反映了从零搭建测试框架的典型挑战（依赖缺失、配置差异、Docker 调试等），所有问题均已解决，未影响测试进度，可作为后续项目的经验参考。
